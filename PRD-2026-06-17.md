# OpenClaw — Product Requirements Document
**Date:** 2026-06-17  

---

## 1. What It Is

OpenClaw is a self-hosted, multi-channel AI agent that runs on a VPS. Never to be launched locally, VPS only. You message it via Telegram; it thinks, uses tools, and replies — maintaining one consistent persona across channels.

The first skill shipped on top of the agent runtime is a **Family Movie & Anime Recommender**: a conversational assistant that learns what your household likes, tracks what you've watched, and suggests age-appropriate titles.

---

## 2. The Host Platform (OpenClaw Core)

### Runtime
- **Node.js / TypeScript** on Ubuntu 24.04, inside Docker.
- Code is edited locally and pushed; the agent runs exclusively on the VPS. API keys never touch the local machine.

### LLM Stack
| Complexity | Model |
|---|---|
| High | Claude Opus 4.6 |
| Default / general | GPT-5.4 |
| Context summarization | GPT-5.4 |
| Complexity classifier | GPT-5 Mini |

If a provider's balance hits zero, OpenClaw switches to the other and alerts the user over Telegram.

### Architecture
Three layers: **Gateway** (WebSocket) → **Agent Runtime** (sessions, tools, memory, context) → **Channel connectors** (thin MCP servers, one per platform).

### Sessions
Session = task, not time window. A GPT-5 Mini classifier decides whether a new message continues the current session or starts a new one. Sessions archive after 7 days of inactivity (or on `/end`), producing a GPT-5.4-written summary that is embedded into Qdrant for future retrieval.

### Lane Queue
Each conversation has its own lane. A new message mid-response cancels the current run and restarts with the updated history (**steering**). Lanes never block each other.

### Memory
- **SQLite** — active session history.
- **Qdrant** — semantic + keyword (70/30 hybrid) search across archived sessions.

### Context Management
Compaction triggers at ~80% of the model's context window. GPT-5.4 summarizes the oldest chunk. Any single tool result is hard-capped at 30% of the window.

### Tools (core, always on)
Browser (Playwright/Chromium, headed via Xvfb), Bash, memory search, cron, heartbeat (proactive task resumption every 5 min).

### Sub-agents
Complex tasks spawn parallel sub-agents. Sub-agents cannot spawn their own (depth ≤ 1, enforced at runtime). Sub-agent sessions persist to SQLite with `parent_session_id` and are re-queued on restart.

### Security
All external content is wrapped in an untrusted-context boundary before the model sees it. Prompt injection and Unicode homoglyph scanning. OpenClaw runs on its own isolated Docker network.

## 3. Skill Framework

A shared **Skill layer** (`src/skills/`) makes domain capabilities composable:

- **`Skill` contract** — `{ name, description, examples, tools, executeTool, systemPromptFragment, migrate }`.
- **Skill registry** — collects registered skills; exposes their tools and prompt fragments to the runtime.
- **Activator** — decides which skills are live for a session. Stem keyword gate: a message activates a skill when its tokens **prefix-match** one of the skill's example phrases (token sequence). Anchored at word start, so the stem `фильм` matches inflected forms like `фильмах`/`фильмов`, but — unlike a bare substring match — `смотр` does not match `осмотр`. **Sticky + additive**: a skill added to a session stays (cap ~2–3, LRU eviction) rather than being swapped out. Persisted on the session so the active set survives restarts.
- **Runtime wiring** — at turn assembly, only `core tools + active skills' tools` are injected, plus their `systemPromptFragment`. Core tools are always present.

Movies is the first consumer of this framework; future skills (e.g. travel) reuse it without changes to core.

## 4. Movies Skill

### 4.1 Summary

Conversational movie and anime recommender for a family. No slash commands — the agent infers intent from natural language and calls the right tool(s). Runs **inline in the main agent** (not as a sub-agent) to preserve the single persona and full shared context.

### 4.2 Key Design Principles

- **Content-based learning, not ML.** Each viewer has a weighted feature profile (genres, themes, tropes, tone, pace, runtime, source type) updated after every feedback event. Transparent, explainable, useful from the first title.
- **Tropes outrank genres.** Tropes predict a child's reaction better ("underdog becomes hero", "wise mentor"). They're extracted by the LLM from synopsis/reviews, normalized against an internal dictionary, and weighted 3× vs genres 1.5×.
- **Stateless tools.** Every tool is a pure function of `(args, recommender.db)`. No in-memory per-conversation state — steering is always safe.
- **Separate database.** Learning state lives in `recommender.db` (not the session store). OpenClaw's context compaction never touches it.
- **Single household.** No multi-tenancy; no `household_id` threading through tools.

### 4.3 Sources

| Source | What it provides |
|---|---|
| TMDB | Movies and series — metadata, age ratings, runtime, genres. Multi-search (movies + series) when the media type is unknown |
| Jikan (MyAnimeList) | Anime — tags, themes, age ratings, seasons |
| LLM (internal) | Tropes extracted from synopsis; normalized to trope dictionary |

All sources normalize into one `NormalizedTitle` shape so scoring is source-agnostic. Adding a new source = implementing `SourceAdapter`.

**Resolution order:** cache → primary adapter (Jikan if `mediaType=anime`, else TMDB multi-search across movies *and* series) → Jikan fallback (if primary was TMDB and returned nothing) → **unresolved**. The household `language` (e.g. `ru-RU`) and any stated release `year` are forwarded to the source, and titles are passed in the user's original language (never translated). Results are ranked — exact/normalized title match, then year match, then popularity — and clip/compilation/"greatest moments" decoys are filtered out, rather than blindly taking the first hit. When nothing resolves, the tool returns a *could-not-resolve* outcome and the agent asks the user (for a year or original title) instead of fabricating a metadata-less stub that would be logged and learned from. A bare query may reuse the loose title cache, but an explicit `year`/`mediaType` (a correction) bypasses it and re-resolves from source so a stale match can't stick.

### 4.4 Data Model (separate `recommender.db`)

14 tables:

| Table | Purpose |
|---|---|
| `household` | Singleton settings (tz, language, onboarded flag) |
| `user` | Viewers — name, birth_date or age_static + age_recorded_at |
| `user_preference` | Per-viewer feature weights (dimension × value × weight) |
| `user_constraint` | Hard rules: triggers, max_runtime, excluded tropes/themes, etc. |
| `title` | Unified title cache (source-agnostic) |
| `trope_dictionary` | Canonical trope ids + aliases for LLM normalization |
| `taxonomy_map` | Maps TMDB/Jikan source terms to canonical genre/theme ids |
| `watch_event` | A viewing record |
| `watch_event_viewer` | Join table with `age_at_watch` frozen at watch time |
| `feedback` | Rating per viewer per watch; `applied_to_profile` idempotency guard |
| `watchlist` | Favorites (loved) and wishlist (want to watch) |
| `suppression` | Do-not-recommend entries (title/trope/theme/genre) |
| `recommendation_log` | Everything shown, with score, reasons, and outcome |
| `action_log` | Previous state snapshots for undo |

Current age is always computed (never stored). Viewer ages at past watches are frozen and never recomputed.

### 4.5 Tools (8)

| Tool | What it does |
|---|---|
| `recommend` | Generate, filter, and score candidates for solo or joint watch |
| `log_watch` | Record a viewing; resolves title via cache then TMDB/Jikan. Accepts an optional `year` to disambiguate sequels/remakes |
| `add_feedback` | Record a rating and apply it to the viewer's profile |
| `manage_viewers` | Add / edit / list / remove viewers |
| `manage_taste` | Set preferences (NL), add to watchlist, suppress titles/themes/tropes. `set_preferences` with loved titles returns a **preview** (resolved as `confident`/`ambiguous`/`unresolved`) and writes nothing until confirmed |
| `show` | Display favorites, watch history, or taste profile |
| `setup` | First-run household + viewer creation from a free-text line |
| `undo_last` | Revert the last recorded action via `action_log`; works across sessions (household-scoped) |

One user message can trigger multiple tools (e.g. `log_watch` + `add_feedback`).

**Confirm before committing watched+loved.** When a user reports titles they "watched and loved", `set_preferences` only *previews* the resolved matches — `Title (year)`, with alternatives for ambiguous ones and a list of anything that didn't resolve. Free-text preferences apply immediately, but the watch + `loved` writes happen only after the user confirms, at which point the agent commits each title via `log_watch` + `add_feedback(rating=loved)`. Numbered ranges (`"Star Wars 1-9"`, `"seasons 1-3"`) are expanded to their installments in the preview and committed one `log_watch` per installment. Every committed write is reversible — results carry an undo affordance and the agent surfaces a one-tap "↩️ Отменить".

### 4.6 Recommendation Logic

1. **Generate** — build a source query from the viewer's top features + any explicit constraints; pull from TMDB/Jikan; normalize and cache.
2. **Filter (hard)** — remove anything that violates active constraints, suppressions, already-seen (default), or recently dismissed. **No automatic age ceiling**: titles are never hard-dropped by their official rating. Age-appropriateness is a *soft preference* applied by the model at presentation time (each candidate carries `age_rating`), which leans toward and frames titles for the youngest viewer but may still surface more grown-up titles with a flag. The only age-based hard filter is an explicit user-set `max_age_rating` constraint.
3. **Score** — `Σ effectiveWeight(profile, feature) × dimension_multiplier`. Multipliers: tropes ×3, themes ×2, genres ×1.5, source_type ×1. Small bonus for external rating. Joint watch: `Math.min` across viewers for each weight (intersection of positives, union of negatives).
4. **Log** — every shown candidate goes to `recommendation_log` with score and reasons.

### 4.7 Learning

After each `add_feedback` (guarded by `applied_to_profile` for idempotency):
- `loved` → +0.3 on title's features; `disliked` → −0.3; `abandoned` → −0.6 (tropes −0.8).
- `tags` like `too_long` target specific constraints (e.g. lower preferred runtime), not blanket genre penalties.
- A `trigger:`-prefixed tag creates or strengthens a `user_constraint`.
- **Age decay** at read time: exponential (0.85/year) based on `age_at_watch` vs current age. Older signals from when the viewer was younger matter less.

### 4.8 Trope Extraction

1. LLM is given the title's synopsis and returns `[{ phrase, confidence }]`.
2. Each phrase is matched against `trope_dictionary` aliases.
3. High-confidence unmapped phrases get new dictionary entries (slugified canonical id).
4. Low-confidence unmapped phrases are dropped — never stored raw.
5. Only canonical ids are persisted on `title.tropes[]`.

### 4.9 Presentation (Telegram)

**Text cards** (v1):
```
🎬 1. Кунг-фу Панда (2008) · 6+ · 92 мин · ⭐7.6 · TMDB
Динамичное приключение с юмором — как «Зверополис», который зашёл.
Тропы: неудачник-герой · мудрый наставник · комедийный экшен
Почему: совпадение 87% — ✓наставники ✓комедийный экшен ⚠чуть длиннее обычного
```

**Inline buttons** (Phase 9, shipped): recommendation responses auto-attach 👍 / 😐 / 👎 / ✋ / ➕ inline keyboards. Button taps are routed as callback_query → NL text → agent. Posters sent via `sendPhoto` when `poster_url` is available. Text path remains as fallback.

**Ratings from NL**: "1 зашло, в середине заскучал" → the agent parses ratings and tags via the skill's `systemPromptFragment` and calls `add_feedback`; inline buttons (`buttons.ts`) map taps to the same NL shorthand.  
Negative rating → one follow-up: "страшно / скучно / длинно?" → tags.

### 4.10 Interaction Flows (all 13 implemented and tested)

1. First-run setup — one free-text line creates household + multiple viewers.
2. Manage viewers — add, list, edit (including ages), remove with confirmation.
3. Set / extend preferences — from loved titles + NL; updateable later in free text. Loved titles are previewed as `Title (year)` and committed as watched+loved only after the user confirms; numbered ranges expand to their installments.
4. Request recommendations — solo or joint; refineable in-thread ("покороче", "без драконов", "аниме сегодня").
5. Add to favorites — from rec or manually; manual add asks watched vs wishlist.
6. Review and rating — NL parses title, viewer, rating, tags; abandoned = stronger negative.
7. Watch history — chronological with rating and age-at-watch.
8. View/explain profile — human-readable taste + age evolution.
9. Suppress — "Тимур перерос фей" → no-recommend without rating.
10. Joint watch — intersected positives, union of stop-lists; age-appropriateness softly biased toward the youngest viewer (no hard ceiling).
11. Quick mode — single answer, hard runtime limit.
12. Seasonal/thematic — theme filter on top of profile ranking.
13. Correct a mistake — undo last record via `action_log`, including across sessions (a new chat can revert the previous session's last action); confirm-before-commit also catches wrong matches before they reach the DB.

## 5. What Is Not Included

- **Streaming availability** — no "where to watch" lookup.
- **Web UI** — Telegram only.
- **TVTropes scraping** — legal/brittle; tropes are LLM-extracted instead.
- **Multi-tenancy** — single household; one OpenClaw instance = one family.
- **Trained ML model** — content-based profile weights only.
- **WhatsApp / Slack / Discord channels** — connectors exist in the framework but are not wired.

- Profile, learning, decay
- Recommendation engine (all filter types, ranking, joint-watch)

- 13 end-to-end scripted conversation flows

Card rendering and NL rating/tag parsing are now driven by the skill's `systemPromptFragment` (model-side), not standalone `presenter`/`nl-parser` modules.
